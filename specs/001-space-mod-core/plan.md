# Implementation Plan: SpaceReloaded — космическая программа в Minecraft

**Branch**: `001-space-mod-core` | **Date**: 2026-07-07 | **Spec**: [spec.md](./spec.md)

**Input**: Feature specification from `/specs/001-space-mod-core/spec.md`

## Summary

Fabric-мод для MC 26.2+ (деобфусцированный код, официальные имена Mojang): герметичные базы с событийным 26-направленным flood fill, многоуровневая промышленность с энергетикой, ракеты произвольной формы, собираемые BFS-сканом в единую сущность с физически корректным расчётом ЛТХ (Циолковский, центр масс, моменты), орбитальное измерение со станциями, планеты как data-driven измерения, возврат через физический Lander + ISRU, орбитальная кинетическая пушка с межпространственной баллистикой и chunk ticket'ами. Архитектура — двухмодульная: чистое вычислительное ядро (`core`) + Fabric-адаптер (`mod`), по паттернам, верифицированным в исходниках Create / Ad Astra / Advanced Rocketry (см. [research.md](./research.md)).

## Technical Context

**Language/Version**: Java 25 (Temurin 25.0.3; `options.release = 25`)

**Primary Dependencies**: Minecraft `com.mojang:minecraft:26.2` (unobfuscated), Fabric Loader 0.19.3, Fabric API 0.154.x+26.2, Loom 1.17 (`net.fabricmc.fabric-loom`), Gradle 9.5.1

**Storage**: NBT-персистентность сущностей и блок-сущностей; палитровая сериализация структур ракет; датапак-реестры (Codec) для планет/материалов/топлив

**Testing**: JUnit 5 (модуль `core`, обязательно); Fabric gametest для интеграционных сценариев (по возможности в 26.2)

**Target Platform**: клиент + выделенный сервер (Linux/Windows), одиночная и мультиплеер

**Project Type**: Minecraft-мод, многомодульный Gradle-проект (core + mod)

**Performance Goals**: ≤1 мс/тик логики мода в главном потоке; пересчёт зоны 10k блоков ≤500 мс в фоне; сборка ракеты 500 блоков ≤100 мс; TPS ≥19.5 при 10 активных зонах

**Constraints**: главный поток — единственный, кто трогает мир; фоновые расчёты только по снапшотам; chunk ticket'ы с гарантированным освобождением; рендер только через Blaze3D (в 26.2 экспериментальный Vulkan, сырой GL запрещён)

**Scale/Scope**: v1 = 7 user stories, 2 новых измерения (орбита, Луна), ~15 блоков/станков, ~10 материалов, 3 типа полётных объектов (ракета, Lander, снаряд)

## Ключевые проектные решения

### D1. Threading model (принцип IV, FR-005)

```
[Main thread]                        [Sealing executor (1-2 daemon-потока)]
 событие блока → ZoneManager
   └─ snapshot: палитровые копии      
      затронутых секций (дёшево,  ──▶  GasFloodFill.run(VoxelView snapshot,
      O(палитра), не O(блоков))          origin, radius=Chebyshev, 26 dirs)
                                          │ early-exit на первой утечке
 server.execute(результат)  ◀──────────  SealingResult
   └─ ZoneManager применяет статус,
      декомпрессия, партиклы
```

- Снапшот: `SectionSnapshotter` собирает копии `PalettedContainer` секций, пересекающих куб радиуса (для R=32 это максимум 5×5×5 секций… по факту куб 65³ → 5³–6³ секций). Копия палитрового контейнера — O(размер палитры), укладывается в бюджет ≤1 мс.
- Проходимость газа резолвится в фоне: BlockState из палитровой копии + предикат (воздух / тег `#spacereloaded:airtight` / состояние двери / полнота коллизии). Теги неизменяемы между перезагрузками данных — читать из фонового потока безопасно; на perезагрузке данных все зоны инвалидируются.
- Очередь пересчётов ограничена (FR-051): один активный пересчёт на зону, коалесинг событий (burst поломок = один пересчёт), глобальный лимит параллельных задач.
- Оптимизация из Ad Astra: long-упаковка позиций + fastutil-подобные структуры (в `core` — свои простые аналоги на `long[]`-очереди, чтобы не тянуть зависимости; профилировать позже).

### D2. Sealing-семантика (FR-001…004)
Статусы результата разделены (в отличие от плагина-прототипа): `SEALED` (объём замкнут), `LEAK` (вышли наружу через дыру — есть точки утечки), `UNBOUNDED` (достигнут лимит радиуса — «слишком большой/открытый объём»). 26 направлений и чебышёвский радиус переносятся из RoomCheckerPlugin как есть (принцип VIII). Диагностический режим (по запросу игрока) не делает early-exit и возвращает точки утечки для подсветки.

### D3. Rocket assembly & физика (FR-020…026)
- **Скан структуры**: BFS по 6 граням (структурная связность — блоки держатся гранями; 26-направленность — только про газ). От командного модуля, по тегу `#spacereloaded:rocket_parts`, лимит из конфига (default 4096). Посторонний блок в связке → ошибка с позицией.
- **Свойства деталей**: датапак-реестр `part_properties` (Codec): масса кг, роль (engine/tank/hull/seat/gyro), тяга Н, Isp с, ёмкость л, тип топлива. Ванильные блоки в структуре запрещены v1 (кроме декоративных из белого списка с массой по умолчанию).
- **ЛТХ** (`PerformanceCalculator`, core): m_dry=Σm; CoM=Σ(m·r)/Σm; тяга F=Σf_i; центр тяги = Σ(f_i·r_i)/Σf_i; момент инерции — точечные массы: I=Σ m_i·|r_i−CoM|² по осям (упрощение: блоки как точки в центрах — погрешность <8% на блочных решётках, задокументировано); TWR=F/(m₀·g); Δv=Isp_eff·g₀·ln(m₀/m₁), Isp_eff — средневзвешенный по расходу.
- **Полёт** (`FlightIntegrator`, core): полу-неявный Эйлер на 20 Гц; уравнение Мещерского (масса убывает ṁ=F/(Isp·g₀)); крутящий момент τ=Σ(r_i−CoM)×f_i; гиродины дают компенсирующий момент до лимита, потребляя энергию; ориентация — кватернион, v1 ограничивает свободу тангажом+креном (без рысканья) для упрощения коллизий.
- **Entity** (mod): паттерн Create — `Map<локальная позиция → BlockState+NBT>`, палитровая NBT-сериализация; инвентари/баки консолидируются менеджером хранилищ. Разборка восстанавливает блоки с содержимым.
- **Коллизия v1**: пока ракета вертикальна — составной AABB; при наклоне — OBB-проверка для пассажиров (перенос в локальные координаты, вектор разделения — паттерн ContraptionCollider) и семпл-проверка углов корпуса против террейна для крушений. Полный OBB-против-мира — вне scope v1.
- **Пассажиры**: сиденья, `positionRider` через локальный вектор → мировой (паттерн Create).

### D4. Рендер ракеты (FR-024) — зона наибольшего технического риска
26.2: Vulkan experimental, обратный depth buffer, только Blaze3D. План: клиентский `RocketMeshCache` тесселирует BlockState-модели структуры один раз в вершинные буферы по слоям и воспроизводит их с матрицей трансформации сущности. Прототипируется ПЕРВЫМ в M3 (спайк): если стабильного пути через Blaze3D нет — временный fallback: поблочная отрисовка через `BlockRenderManager`-аналог каждый кадр (медленно, но верно), оптимизация после стабилизации Vulkan-пайплайна Mojang'ом. Пересборка меша — только при изменении состава структуры.

### D5. Измерения и переходы (FR-030…031)
- Датапак-измерения: `spacereloaded:earth_orbit` (void, звёзды, низкая гравитация), `spacereloaded:moon` (свой ChunkGenerator: реголит/базальт, кратеры).
- Реестр `spacereloaded:planets` (Codec, паттерн Ad Astra): gravity, atmosphere{present, breathable, pressure}, temperature, solarEfficiency, orbitDimension, deltaVToOrbit, parentBody.
- Гравитация — через Fabric API модификации атрибутов измерения (26.1+), не миксинами.
- Переход: контроллер полёта следит за высотой/скоростью; при достижении порога — телепорт сущности со структурой и пассажирами (ticket'ы на обеих сторонах, см. D6); недобор Δv — баллистическое падение.
- Станции: одна орбита = одно измерение; плоты станций на квадратно-спиральной сетке (паттерн Advanced Rocketry), выделение плота — блоком-якорем станции.

### D6. Chunk tickets — `TicketLease` (FR-026, FR-034, FR-043)
Обёртка над ticket-API: `TicketLease.acquire(world, chunkPos, radius, ttlTicks, owner)` — регистрирует кастомный `TicketType`, хранит слабую запись владельца, автопродлевается активным владельцем и **гарантированно** освобождается: (а) явным `close()`, (б) по TTL, (в) при смерти владельца, (г) при выгрузке мира. Утечка ticket'а логируется как ошибка. Все межпространственные операции идут через lease: полёт ракеты (чанки под ракетой), упреждающая загрузка цели снаряда (по времени полёта), стыковка, посадка Lander.

### D7. Энергия (FR-010)
Старт M2 с проверки: жив ли Team Reborn Energy API для 26.2. Если да — реализуем его интерфейсы напрямую. Если нет — свой минимальный интерфейс (`long`-ёмкости, push/pull, сторонность) в `core/energy` + графовая топология сети с кэшем и событийной инвалидацией; адаптер под TR Energy добавляется при появлении порта. Все машины программируются против нашего интерфейса — решение обратимо.

### D8. Networking
Custom payloads (Fabric networking): `RocketControlC2S` (тангаж/крен/тяга/стейджинг), `RocketStateS2C` (позиция/ориентация/топливо — интерполяция на клиенте), `ZoneStatusS2C` (статус зоны для HUD), `CannonFireC2S` (координаты цели). Сервер валидирует всё (принцип VI): пилот сидит в кресле, дальность цели, боезапас.

### D9. Снаряд пушки (FR-040…044)
`KineticProjectileEntity`: масса + вектор скорости, баллистика в `core/ballistics` (гравитация измерения, опционально сопротивление). Переход орбита→поверхность: телепорт на верхнюю границу с пересчётом координат (масштаб измерений) и сохранением скорости. Упреждение: при выстреле вычисляется ETA и точка удара; lease на чанки цели берётся за N секунд до прибытия. Удар: кратер из кинетической энергии E=½mv² → радиус разрушения ~ E^(1/3) (масштабирование энергии взрыва, обоснование в коде), кастомное разрушение блоков по прочности + импульс сущностям. Персистентность: полный вектор состояния в NBT.

### D10. Жизнеобеспечение игрока (FR-006)
Компонентный скафандр (шлем с запасом O₂, баллоны) — проверка каждые 0.5 с: сущность в небезопасном измерении вне герметичной зоны и без O₂ → урон вакуума. Членство в зоне — O(1) по `Set<long>` зон (см. D1).

## Testing (принцип VII)

- `core`: JUnit 5. Обязательные сьюты: `GasFloodFillTest` (замкнутая комната; диагональная щель = LEAK; лимит радиуса = UNBOUNDED; дверь открыта/закрыта; полость 10k — производительность), `PerformanceCalculatorTest` (Δv против аналитического Циолковского с точностью 1%; CoM/центр тяги; предупреждения TWR<1), `FlightIntegratorTest` (вертикальный подъём против аналитики; асимметричная тяга даёт ожидаемый момент; выгорание топлива), `BallisticsTest` (время падения/точка удара против аналитики), `TicketLease`-семантика тестируется в mod-слое.
- `mod`: Fabric gametest, если работоспособен в 26.2 (проверка в M1): сборка структуры 3×3, sealing сценарий с дверью; иначе — ручные сценарии из quickstart + `/spacereloaded debug`-команды.
